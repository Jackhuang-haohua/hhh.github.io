react生命周期

react的生命周期为啥这么多呢...

按照顺序介绍用途，关于版本变迁、底层原理之类的就不介绍了

一、挂载阶段

constructor

　完成数据的初始化；
　可以在这里三两件事：

  • 定义数据，如this.state={}
  • 接收数据，props和context，必须放在super()之后，super是继承相关，后面的文章再讲
  • 绑定事件函数的this，this.handleChange = this.handleChange.bind(this)，是函数的this指向本组件实例

static getDerivedStateFromProps

　在挂载和更新阶段都会调用；
　名字翻译过来是接收参数，可以代替之前的componentWillReceiveProps；
　接收两个参数(nextProps, prevState)，即比较旧的state和新的props，可以用作派生状态；

派生状态

　子组件接收父组件的n并派生出a：

　如果没有getDerivedStateFromProps的话，子组件中的a永远是1，因为constructor只会执行一次，接收一次props后就不会更改了；
　但getDerivedStateFromProps每一次更新都会执行，重新赋值；

　跟componentWillReceiveProps类似，比较前后的props，再赋值。
　注意：子组件中任何一个自定义方法执行，都会触发getDerivedStateFromProps

render

在这个生命周期函数里执行渲染操作，主要有以下类型：

  • 原生的DOM，如div
  • React组件
  • Fragment（片段）
  • Portals（插槽）
  • 字符串和数字，被渲染成text节点
  • Boolean和null，不会渲染任何东西

componentDidMount

　一个阶段只执行一次，主要用来执行异步操作，比如请求数据，定义canvas，svg，定时器，实例化等操作；
　在这里请求数据和挂载节点同时进行，应该是不能获取节点的；虽然之前实验好像可以获取某些节点，但是最好别在这里这么做；
　比如swiper的实例化，就不能在这里进行，因为节点还没加载完毕；

二、更新阶段

经过console.log的实验，更新阶段会在以下三种情况触发：props改变、setState()、forceUpdate

static getDerivedStateFromProps

　前面已经写过了

shouldComponentUpdate

  • 询问组件是否进行更新操作，默认返回true，代表组件要render更新了
  • 有两个参数nextProps和nextState，表示新的属性和变化之后的state
  • 注意当我们调用forceUpdate并不会触发此方法

render

　前面也写过了

getSnapshotBeforeUpdate

  • 代替componentWillUpdate，
  • 这个方法在初步render之后，componentDidUpdate之前调用，
  • 有两个参数prevProps和prevState，表示之前的属性和之前的state，
  • 这个函数有一个返回值，会作为第三个参数传给componentDidUpdate，如果你不想要返回值，请返回null，不写的话控制台会有警告
  • 作用是读取当前某个DOM节点的状态，并在进行相应的处理；此DOM可以与 componentDidUpdate 中的一致

componentDidUpdate

  • 真实dom已经挂载完，该方法在getSnapshotBeforeUpdate方法之后被调用，
  • 有三个参数prevProps，prevState，snapshot，分别表示之前的props，之前的state，和snapshot。第三个参数是getSnapshotBeforeUpdate返回的
  • 在这个函数里我们可以操作DOM，拿到上面的三个参数，和发起服务器请求，还可以setState，但是注意一定要用if语句控制，否则会导致无限循环
