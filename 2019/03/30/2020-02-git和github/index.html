项目中git和github的使用

一、什么是git和github

git

  • 开源的分布式版本管理器，一个用来管理我们文件夹的”软件”
  • 可以将代码托管到远程
  • 可以进行版本的管理

github

  • 是一个远程的代码托管平台 / 代码仓库
  • 依靠 git 进行代码的上传

二、git的使用

安装

  • 官网下载
  • 检测：git –version

配置

  • 随便在一个地方，鼠标右键，点击git bash here；
  • 输入指令git config --global user.name "你的名字"，设置名字，名字必须是英文字母；
  • 输入指令git config --global user.email "你的邮箱"，设置邮箱；

常用指令

指令 描述
git init 初始化文件夹，生成.git文件，告诉git这个文件夹是一个仓库
git status 查看暂存区
git add 文件名 把文件添加到暂存区
git add 文件夹名称/ 把文件夹添加到暂存区
git add –all 把所有文件夹添加到暂存区
git reset HEAD – 文件名 拉回单独一个文件到工作区
git reset HEAD – 文件夹名称/ 拉回单独一个文件夹到工作区
git reset HEAD – . 拉回暂存区所有文件到工作区
git commit -m “本次版本的说明(中英文无所谓)” 把暂存区所有代码形成一个历史版本
git clone 克隆仓库
git log 查看当前已经形成的历史版本
git reset –hard ‘id’ 回退历史版本，’id’即git log中，commit后面一串字符

三、上传github

假设我们现在有一个项目文件夹shop，里面有css、html、JavaScript等文件夹；

  • 注册github账号，点右上角 + 号，New repository新建仓库，必填项就一个仓库名字；
  • 如下图，点击HTTPS，复制后面的链接；

  • 在shop文件夹里，右键Git Bash Here；
  • 输入指令git init，文件夹里多了个.git隐藏文件夹；
  • git remote add origin HTTPS，HTTPS即为第二步中的链接，这里是添加仓库地址；
  • git add .把所有文件转到’暂存区’；
  • git commit -m "这里是你的描述"，把’暂存区’文件形成历史版本；
  • git push -u origin master，上传仓库，origin即为之前存的仓库地址；
  • 之后上传只需要git push;

四、git创建分支

分支方便团队协作，看起来比较抽象，其实试一下就知道怎么回事了；
我们git管理文件夹后，默认主分支为master，在Git Bash Here也可以看到后面的(master)；
下面一步步来：
dev分支：

  • git branch dev，创建master的dev分支
  • 在dev分支里写代码

list分支：

  • git branch list，创建dev的list分支
  • 在list分支里写内容
  • git add .加到到暂存区
  • git commit -m "描述"，创建了历史版本，此时，list拥有dev的内容，dev没有list的内容
  • 小提示：git branch -D list，删除list分支

login分支：

  • git checkout dev，转到dev分支
  • git branch login，创建dev的login分支
  • 在login分支里写内容
  • git add .
  • git commit -m "描述"，此时，list和login同为dev的分支，但彼此的内容不相同

合并：

  • git checkout dev，转到dev分支
  • git merge login、git merge list，把login和list合并到dev身上
  • git checkout master，转到master主分支
  • git merge dev，合并到master主分支
  • git add .
  • git push，上传到仓库

上传本地分支
git push origin list可以把本地分支推送到远程分支，一般主分支不允许普通员工更改
删除本地分支
git branch -d list先转到master主分支再删除
删除远程分支
git push origin :list先转到master分支，再执行命令删除

这么一来，master分支有dev，dev分支里有list和login，切换到分支再写代码，就不会混乱了。

五、分支的命名规则

  • ‘master’是主分支，不接受代码上传只接受代码合并，一般只接收dev；
  • ‘development / develop / dev’是主开发分支不接受代码上传只接受代码合并，一般只接收主开发分支；
  • ‘feature-xxx(feature-login / feature-list / …)’是主开发分支，从develop分支上拓展，是一个个项目功能分支；
  • ‘feature-xxx-fix’bug修复分支，从主开发分支上分出来，主要进行代码的 bug 修复，修复后合并到主开发分支；
  • ‘hotfix-xxx’临时分支名称, 项目运行的时候出现了必须要立刻解决的问题开辟，直接在 master 分支上开辟一个 hotfix 分支,解决完毕后直接合并到 master 分支上；

六、git冲突问题

假设A和B同时clone了github上的项目，在同一个分支里编写代码；
编写完后，A和B都需要上传，A上传成功后，B就无法上传了；
因为github认为有重复项，不知道应该保存哪个；
此时B需要git pull/git pull origin master拉取整个仓库，再git push，会询问你保留哪个版本，就可以上传了。

七、GitHub Desktop

每次上传github都需要git操作，好像有点麻烦，其实git有可视化桌面工具 — GitHub Desktop；
用法很简单，File->New repository新建仓库、Branch->New branch新建分支、Branch->Merge into current branch合并分支，与上面的指令一致，就不多介绍了。

八、github冲突问题

  • 假设A和B同时clone了github上的项目，在同一个分支里编写代码；
  • 编写完后，A和B都需要上传，A上传成功后，B就无法上传了；
  • 因为github认为有重复项，不知道应该保存哪个；
  • 此时B需要git pull/git pull origin master拉取整个仓库，再git push，会询问你保留哪个版本，就可以上传了。

九、分支案例

之前的分支问题又研究了一下，记录下来；
假设现在项目已经写了一半了，但是还没创建自己的分支，

  • 先把现在写的复制一份，起名为”design-abc”，其中abc是分支名，可以随便起
  • 新建一个空文件夹
  • git clone "https://github.com/gzh51914/design" ，从公共仓库克隆，得到design文件夹
  • 进入design文件夹
    • 右键Git bash here，第一行最右边有(msater)
    • 输入git branch abc，创建了master下的abc分支
    • 输入git checkout abc，跳转到abc分支，发现右边变成了(abc)
    • 把之前”design-abc”里的文件全部复制进来
    • 输入git add .
    • 输入git commit -m "修改abc分支" 或 git commit -m "修改abc分支" --no-verify
  • 此时vscode中yarn serve，8000端口出现的是你自己写的内容
  • 在git窗口中git checkout master，8000端口变成了公共仓库里的内容，就代表你的分支成功了
  • 然后输入git checkout abc转到你的分支
  • 输入git push origin abc上传分支
  • 到github仓库查看，点击左上的Branch，会发现有你自己的分支，并且master分支和abc分支内容不相同

此后，写代码时，先看看是不是自己的分支，Git bash here右边，或者vscode左下角，都可以查看分支；

当你写完一个阶段，想push到远程仓库了

  • git pull origin abc
  • git add .
  • git commit -m “上传abc分支”
  • git push origin abc

当你想取得远程仓库的内容

  • git pull origin master 或 git pull origin abc
